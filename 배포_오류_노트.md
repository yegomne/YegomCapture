# 🚀 이과장의 인프라/배포 오류 노트 & 오답 노트

## 🚨 사건 1: GitHub Push Protection 락(Lock) 발생 (GH013 에러)
* **원인**: `release.ps1` 스크립트 파일 내부에 깃허브 개인 액세스 토큰(PAT, `ghp_...`)이 평문(하드코딩)으로 저장된 상태에서 `git push`를 시도함. 깃허브 보안 서버가 이를 감지하고 레포지토리 푸시를 전면 차단함.
* **해결책**: 스크립트 파일에서 PAT를 삭제하고 `git commit --amend`로 과거 기록까지 깨끗하게 덮어쓴 뒤 푸시 성공.
* **향후 방지책**: 토큰이나 API 키 같은 민감 정보는 절대 코드나 파일에 하드코딩 되어 Git에 올라가지 않도록 주의할 것. 대신 로컬 환경변수나 GitHub Actions의 Secrets 기능을 써야 합니다.

## 🚨 사건 2: 랜딩페이지 다운로드 링크 404 에러 (시차 발생)
* **원인**: 배포 버전 업데이트(`1.0.1` -> `1.0.2`)를 진행하면서, 깃허브 릴리즈 서버에 `v1.0.2` 실행 파일(.exe)이 정상적으로 업로드되지도 않았는데(인증 에러로 실패), 랜딩페이지(`index.html`)의 다운로드 주소 먼저 `v1.0.2`로 고쳐서 푸시해 버렸음. 이로 인해 유저들이 다운로드를 눌렀을 때 파일이 없는 404 Not Found 에러 발생.
* **해결책**: 백그라운드 환경 변수에 PAT를 넣고 다시 `gh release create`를 성공시켜 `v1.0.2` 파일을 무사히 서버에 넣음.
* **향후 방지책**: 
  1. 다운로드 링크(프론트엔드) 업데이트는 **반드시 릴리즈 파일(백엔드/서버) 업로드가 성공적으로 완료된 것이 확인된 100% 이후**에 진행할 것. 즉, "업로드 완료" 사인을 먼저 확인하라!
  2. 스크립트 자동화 시 `gh auth status` 또는 파일 업로드 성공(`%errorlevel% eq 0`) 검증 절차를 이중으로 확인할 것!
